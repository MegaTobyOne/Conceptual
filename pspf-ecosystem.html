Australian PSPF tool ecosystem

Start small, stay traceable, speak the PSPF language.

PSPF is the Australian Government's Protective Security Policy Framework. This ecosystem helps Australian entities assess PSPF requirements and the ASD Essential Eight locally, then share a clear publication-mode view through Explorer.

See how it works Go to the links
At a glance

v0.1 proves the spine before the full ecosystem.

Core keeps the trusted local record. Workshop supports Requirement, Evidence, Action, and Risk authoring in VS Code. Explorer turns a curated bundle into a shareable, read-only publication view. Shop and Pub remain part of the design, but are deliberately deferred until the spine is reliable.

  • 1
    Capture and maintain

    Core and Workshop organise Requirements, Evidence, Actions, and Risks for the v0.1 assessment loop.

  • 2
    Validate and package

    The working data is checked, snapshotted, versioned, and exported into a shareable JSON bundle.

  • 3
    Explore and explain

    Explorer turns the bundle into a simple publication-mode interface for posture, readiness, evidence freshness, and shareable brief copy.

How the ecosystem works together

The ecosystem separates policy source, operational working data, and published sharing views. Authoritative PSPF content comes from Home Affairs, Essential Eight terminology comes from ASD, local working data stays in the toolchain, and shareable views are generated from that data rather than edited by hand.

Source

PSPF guidance

The PSPF site defines the framework, PSPF Domains, requirements, structure and reporting expectations. It is the place to check official wording and the current release.

Working layer

Core and Workshop first

v0.1 captures local assessment state, evidence, actions, and risks. Core is the local store; Workshop is the VS Code authoring surface that talks to Core.

Sharing

Explorer

Explorer consumes a curated JSON bundle and presents the information in a simple static interface. v0.1 is publication mode only; browser-local authoring is deferred.

Products in the ecosystem

Each product has one main job. v0.1 implements only the shortest useful path: initialise a workspace, create assessment evidence, export a bundle, and share an Explorer posture brief.

Core

v0.1

Core is the local system of record. It manages the canonical entity and link model, validation, writer lock, snapshots, bundle import/export, and operational health.

  • Stores the structured PSPF workspace.
  • Runs validation, backup, restore and integrity checks.
  • Exports the bundle consumed by Explorer.

Explorer

v0.1 publication

Explorer is the simple, portable web interface. v0.1 reads a curated bundle, shows posture and readiness, and lets reviewers copy a shareable brief without installing VS Code.

  • Presents posture, gaps and readiness visually.
  • Lets users drill from summary to detail.
  • Shows OFFICIAL: Sensitive and TLP:AMBER+STRICT handling cues.
  • Browser-local authoring and round-trip export are deferred.

High-level instructions

This is the simplest way to explain the v0.1 operating model to someone else. It is intentionally high level so people can see the assessment spine before worrying about implementation detail.

Step 1

Start with the PSPF

Use the official PSPF site to understand the current framework, PSPF Domains, requirements, and guidance. Treat that source as the policy authority.

Step 2

Capture local assessment reality

Use Core with Workshop to record how your environment maps to the framework, including assessment status, linked evidence, actions, and risks.

Step 3

Share the picture

Export a bundle and publish it to Explorer so reviewers can see posture, evidence freshness, gaps, and the shareable brief from summary to detail.

Development model

The implementation plan is deliberately lightweight: one private GitHub Pro repository, pnpm workspaces, VS Code on macOS for the maintainer loop, and GitHub Actions for repeatable checks and releases.

Source

Single monorepo

Core, Workshop, Explorer, shared contracts, docs, ADRs, and schemas live together so one pull request can update the contract and every affected product.

Local loop

VS Code on macOS

The maintainer path uses VS Code, Node LTS, pnpm via Corepack, SQLite WAL checks, Playwright for Explorer, and GitHub CLI for release tasks.

Guardrails

Tests before trust

v0.1 gates include AU-English lint, schema-policy checks, personal-data exclusion, accessibility floor, contract tests, fixture round-trips, and a single-writer lock test.
