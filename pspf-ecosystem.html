Australian PSPF tool ecosystem

Start small, stay traceable, speak the PSPF language.

PSPF is the Australian Government's Protective Security Policy Framework. This ecosystem helps Australian entities assess PSPF requirements and the ASD Essential Eight locally, then share a clear publication-mode view through Explorer.

How the ecosystem works together

The ecosystem separates policy source, operational working data, and published sharing views. Authoritative PSPF content comes from Home Affairs, Essential Eight terminology comes from ASD, local working data stays in the toolchain, and shareable views are generated from that data rather than edited by hand.

Source

PSPF guidance

The PSPF site defines the framework, PSPF Domains, requirements, structure and reporting expectations. It is the place to check official wording and the current release.

Working layer

Core and Workshop first

The current slice captures local assessment state, evidence, actions, risks, Directions, and ISM mappings. Core is the local store; Workshop is the VS Code authoring surface that talks to Core.

Sharing

Explorer

Explorer consumes a curated JSON bundle and presents the information in a static interface with a posture donut, compact status table, collapsible sections, AU dates, ISM coverage, readable relationship tables, local annotations, saved views, and a read-only Plan Lens.

Products in the ecosystem

Each product has one main job. The current repository slice supports a practical validation path: load sample data, review assessment posture, edit key records, create planning views, export a bundle, verify release state, and share an Explorer posture brief. Core

v1.6.0 · Marketplace live
VS Code extension tobyharvey.pspf-core Local-only

Core is the local system of record. It runs entirely inside VS Code, holds the structured workspace in a local SQLite database, and enforces a single-writer lock so only one editing session touches the data at a time.

  • Owns the canonical entity and link model (Requirements, Evidence, Actions, Risks, Directions, ISM mappings).
  • Runs schema-policy validation, snapshots, backup, restore and integrity checks.
  • Exports the curated manifest-led JSON bundle that Explorer consumes.
  • Repository package version is v1.13.0; Marketplace publication is pending a real release run.
  • Required dependency for Workshop — install Core first.

Workshop

v1.6.0 · Marketplace live
VS Code extension tobyharvey.pspf-workshop Depends on Core

Workshop is the day-to-day authoring surface inside VS Code. It opens screen-based views for each entity type, talks to Core through a typed contract, and provides save-and-close editing flows that respect the writer lock.

  • Screens for Requirements, Evidence, Actions, Risks, Directions and ISM mappings.
  • Item-detail webview panels with compact edit tab titles and AU-formatted dates.
  • Highlights stale evidence, gaps, follow-up actions, tags, saved views, Change Records, and planning lenses for the current assessment loop.
  • Triggers the export that produces the Explorer bundle.
  • Repository package version is v1.13.0; Marketplace publication is pending a real release run.

Explorer

publication mode
Static web view Reads JSON bundle Read-only

Explorer is a portable, static web interface that reads a curated bundle exported by Core and renders it for review. It needs no install — reviewers open it in a browser to see posture without touching the source workspace.

  • Posture donut, compact status table and collapsible sections with a Close All control.
  • Action Impact, ISM coverage, saved views, tag filters, Why This Changed, Plan Lens, and readable relationship tables.
  • Short AU dates, compact unresolved IDs, and OFFICIAL: Sensitive / TLP:AMBER+STRICT handling cues.
  • Copyable posture brief for handing to reviewers.
  • Browser-local status, evidence, Actions, Risks, saved views, and import-review round trip are available in the repository build.

High-level instructions

This is the simplest way to explain the current operating model to someone else. It is intentionally high level so people can see the assessment spine before worrying about implementation detail.

Step 1

Start with the PSPF

Use the official PSPF site to understand the current framework, PSPF Domains, requirements, and guidance. Treat that source as the policy authority.

Step 2

Capture local assessment reality

Use Core with Workshop to record how your environment maps to the framework, including assessment status, linked evidence, actions, risks, Directions, and ISM mappings.

Step 3

Share the picture

Export a bundle and publish it to Explorer so reviewers can see posture, evidence freshness, gaps, Action Impact, ISM coverage, relationships, planning focus, change context, and the shareable brief from summary to detail.

Development model

The implementation plan is deliberately lightweight: one private GitHub Pro repository, pnpm workspaces, VS Code on macOS for the maintainer loop, and GitHub Actions for repeatable checks and releases.

Source

Single monorepo

Core, Workshop, Explorer, shared contracts, docs, ADRs, and schemas live together so one pull request can update the contract and every affected product.

Local loop

VS Code on macOS

The maintainer path uses VS Code, Node LTS, pnpm via Corepack, SQLite WAL checks, Playwright for Explorer, and GitHub CLI for release tasks.

Guardrails

Tests before trust

Current gates include AU-English lint, schema-policy checks, personal-data exclusion, accessibility floor, Explorer publication and local-authoring smoke tests, Marketplace dry-run signalling, public version verification, receipt-tag checks, contract tests, fixture round-trips, and a single-writer lock test.
